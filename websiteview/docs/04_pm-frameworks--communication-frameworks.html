1. 🎯 承諾層次框架(commitment framework)— 對客戶承諾前必判斷的 4 層

承諾層次框架(Commit Framework)

Kenny 自萃取自 Platform Science / CONNECTSOURCE 兩案的 PM 框架

核心命題

研發 PM 不是承諾「結果」,是承諾「透明度」。

「七月到的時候,客戶不會是第一個知道『做不到』的人。」

4 層承諾

類型 範例 風險 適用
功能上線日期 「七月上線」 高 研發確定性極高才用
里程碑節點 「六月底給你 MVP 評估結果」 中 階段評估 / 測試版
評估時程 「兩週內告訴你能不能做」 低 需求確認初期
資訊更新頻率 「每月底給進度更新」 極低 任何時期都可承諾

金句

信任不是來自準時,是來自事先預警。

同樣是延兩個月:

  • ✅ 事先預警 → 建立信任
  • ❌ 事後才說 → 摧毀信任

應用到 VMX 實際情境

對 Vinicius(Platform Science)

  • 不講「Custom SDK 七月上線」(高風險)
  • 講「兩週內告訴你 SDK 文件齊全度」(低風險)+「每月底我會給進度」(極低風險)

對 BMS Smartlink

  • 不講「Yawning Q2 上」(高風險,內部測試還不過)
  • 講「Yawning 進入 user TR 階段我們會立刻 sync 給你」(里程碑,中風險)

對 HDFE UI 抱怨(2026-05-06 會議)

  • 不講「九月底正式 deprecate 舊版」(高風險,客戶會抗拒)
  • 用 sunset 流程:Q2 預告 + GA 追蹤 + Q3 9月底正式移除(里程碑 + 資訊更新組合)

跟 KB / Sheet / Roadmap 三層真相關係

  • 三層真相 = 誰看到什麼版本(空間維度)
  • 承諾層次 = 用什麼語法承諾(語言維度)
  • 兩者互補

2. 💬 技術型客戶溝通(tech-client-comm)— Vinicius / Cary 類型客戶處理 SOP

技術型客戶溝通

源自 Vinicius / Platform Science 案例

對立行動清單

❌ 不有效 ✅ 有效
強調功能多強大 提供精準技術文件
只說商業故事 清楚說功能限制
等客戶問才回應 跟上客戶測試節奏
技術問題模糊帶過 主動定義 Next Step + Owner + 時限
讓報價「等一下」 報價作為信任指標,早給

關鍵洞察

「Vinicius 這類技術型決策者,每次互動都在測試 MiTAC 的反應速度與專業度。K265 portal 問題對他不只是 bug — 是『這個合作夥伴值不值得依賴』的評估。」

識別技術型客戶的訊號

  • 對方先看 spec / 文件 / API,後問商業
  • 會主動提技術細節挑戰你
  • 重視回覆速度多於話術
  • 報價 / 時程 modulation 比較敏感(看是否專業)

3. 🔄 內部斷層(internal-comm-gap)— 跨團隊(MAU vs HQ vs RD)資訊不對稱處理

內部溝通斷層

源自 Wendy 案 + 2026-05-06 Sheet/Jira 同步缺口發現

業務端不滿的根因表

業務感受 根因 PM 解法
「不知道功能什麼時候出」 產品端沒主動輸出 roadmap 建立定期 roadmap sync(每季)
「不知道能否承諾」 承諾層次太粗,沒里程碑 提供「確認日期」而非「上線日期」
「功能說好有,但用不了」 原型 ≠ 可交付,差異未傳達 早期清楚說就緒條件
「感覺被蒙在鼓裡」 無常態資訊流,只有被動回應 主動推播,沒進展也要說

核心洞察

業務端的不滿通常是資訊不對稱,不是情緒問題。

PM 有責任主動輸出資訊,即使功能沒進展,也要讓業務端知道「我們知道、目前狀態是 X」。

Sheet ↔ Jira 雙向同步缺口(2026-05-06 發現)

PM 角色升級成「Sheet/Jira sync owner」,要處理兩個方向的落差:

方向 1:Sheet 落後 Jira(Jira 已關 Sheet 沒改)

  • 例:VMX-6754 / 7029 / 7161 / 7162 — 5/6 Q2 Review 前
  • 業務端看 Sheet 以為「拖了 7 個月」,實際 RD 已交付

方向 2:Sheet 領先 Jira(口頭說完成,Jira 沒 close)

  • 例:VMX-7082 / 7233 / 6782 — 5/6 Q2 Review 後新生成
  • 「我做完了」≠ Jira ticket transition

PM 介入動作

每兩週做一次 Sheet × Jira sweep:

  1. 撈 Sheet 顯示 In Process 的 ticket
  2. 比對 Jira 真實狀態
  3. 兩個方向 mismatch 都私訊 owner 確認
  4. 同步後在 Sheet 加 sweep 紀錄

→ 這就是評估期 PM 真實的「沒人扛 = 我來扛」入口。
