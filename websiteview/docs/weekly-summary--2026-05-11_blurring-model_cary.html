Operating Scenarios

Three common patterns based on how Blurring is typically used by SIs.

Scenario A · Default

All inherited from Master

Master is enabled, no individual overrides are set anywhere. Every Fleet and Contract Fleet inherits ON automatically — no additional API calls required.

Scenario B · Selective

Disable specific Contract Fleet

Master remains ON. Your SI calls the API to set Contract Fleet B's individual setting to OFF. Contract Fleets A and C continue to inherit ON; B stays OFF until changed.

Scenario C · On-demand

Trigger blur per video

Your SI's application calls POST /blur with a video ID and fleet ID — for example, when an end user clicks "blur this clip" in your platform. Each call is logged for usage-based reporting.

Built-in Support for Monthly Subscription Reporting

Because every action — both configuration changes and blur triggers — flows through the API, all activity is logged at call level. This provides a richer data source than a simple on/off toggle for the monthly subscription report you raised.
