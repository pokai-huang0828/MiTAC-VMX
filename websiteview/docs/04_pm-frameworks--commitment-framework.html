承諾層次框架(Commit Framework)

Kenny 自萃取自 Platform Science / CONNECTSOURCE 兩案的 PM 框架

核心命題

研發 PM 不是承諾「結果」,是承諾「透明度」。

「七月到的時候,客戶不會是第一個知道『做不到』的人。」

4 層承諾

類型 範例 風險 適用
功能上線日期 「七月上線」 高 研發確定性極高才用
里程碑節點 「六月底給你 MVP 評估結果」 中 階段評估 / 測試版
評估時程 「兩週內告訴你能不能做」 低 需求確認初期
資訊更新頻率 「每月底給進度更新」 極低 任何時期都可承諾

金句

信任不是來自準時,是來自事先預警。

同樣是延兩個月: - ✅ 事先預警 → 建立信任 - ❌ 事後才說 → 摧毀信任

應用到 VMX 實際情境

對 Vinicius(Platform Science)

  • 不講「Custom SDK 七月上線」(高風險)
  • 講「兩週內告訴你 SDK 文件齊全度」(低風險)+「每月底我會給進度」(極低風險)

對 BMS Smartlink

  • 不講「Yawning Q2 上」(高風險,內部測試還不過)
  • 講「Yawning 進入 user TR 階段我們會立刻 sync 給你」(里程碑,中風險)

對 HDFE UI 抱怨(2026-05-06 會議)

  • 不講「九月底正式 deprecate 舊版」(高風險,客戶會抗拒)
  • 用 sunset 流程:Q2 預告 + GA 追蹤 + Q3 9月底正式移除(里程碑 + 資訊更新組合)

跟 KB / Sheet / Roadmap 三層真相關係

  • 三層真相 = 誰看到什麼版本(空間維度)
  • 承諾層次 = 用什麼語法承諾(語言維度)
  • 兩者互補
